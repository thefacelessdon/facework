# V0.0.8 Plan — Efficiency Hints + Operating Polish

Status: Draft
Target: v0.0.8
Plan author: working session, 2026-05-12
Folds into: schema + PROTOCOL.md §9/§10/§11 amendments once executed
Predecessors: OPERATING-LAYER-PLAN.md (trilogy v0.0.5–v0.0.7 — shipped)

## 1) Purpose

Close gaps surfaced by four independent research sources without growing
the protocol's surface area. **All 10 additions are optional declarations
— no new MUSTs, no new artifacts, no new phases.** v1.0.0–v1.3.0 manifests
stay conformant. v1.4.0 adds new optional fields that runtimes that can
use them benefit from, and runtimes that can't ignore.

The research that drove this:

- **Karpathy (Agentic Engineering)** — Software 3.0 = context-engineering;
  verifiability is the lever; sensors-and-actuators framing; outsource
  thinking, not understanding.
- **Fung (Cloud Code engineering org)** — bottlenecks shifted to
  verification + review; explicit permission to kill old processes; code
  is source of truth.
- **Asana (Claude Managed Agents)** — multiplayer agents; agent
  ownership/sponsorship; built-in grader/verifier loops; audit trail.
- **Claude Platform efficiency talk** — prompt caching (80%+ hit rate),
  tool search, programmatic tool calling, compaction, advisor strategy.

The cutline: **declare properties, don't mandate behavior.** The protocol
declares cache affinity, load mode, model preference; runtimes implement
the actual caching, tool-search, model selection.

## 3) Per-addition specification

### 3.1 — `sponsors[]` on Skill

Human(s) accountable for the skill's correctness, memory hygiene, and
lifecycle. Mirrors Asana's "agent sponsors" concept.

```yaml
- id: prospect-qualification
  sponsors: ["harper@face.works"]
  # ... existing fields ...
```

Validator: each entry SHOULD be an email or stable handle. No format
enforcement beyond string non-empty (different orgs use different
identifiers).

### 3.2 — `verifier_skill_id` on Skill

Generalizes the `design-eye-evaluator` pattern. Any output-producing
skill MAY declare a verifier skill that gates its output before the
runtime publishes the result.

```yaml
- id: engagement-delivery
  verifier_skill_id: "design-eye-evaluator"
  # ... existing fields ...
```

Validator: `verifier_skill_id` MUST resolve to a Skill in the same
manifest with `domain: quality` (recommended) and `ownership` of `agent`
or `hybrid`.

### 3.3 — `multiplayer: bool` on Skill

Declares whether the skill can be invoked by multiple users concurrently
and whether memory writes accrue across users (Asana pattern).

```yaml
- id: prospect-qualification
  multiplayer: true
  # ... existing fields ...
```

Default: `false`. Setting to `true` declares that the skill's memory
writes are shared and corrections from user A become visible to user B.

### 3.4 — `direction` on Integration scope items

Adopts Karpathy's sensors/actuators framing. Each scope item in an
integration can declare whether it reads, writes, or both.

```yaml
- id: linear
  scope:
    - operation: read
      direction: sensor
    - operation: write
      direction: actuator
    - operation: comment
      direction: actuator
```

Currently `scope` is a flat list of strings. v1.4.0 allows scope items
to be either a string (back-compat) or an object with `operation` +
`direction`. Both are valid in the same manifest.

### 3.5 — Minimum audit-event spec

Prose section added to PROTOCOL.md §9 (or §12 — see §7.5 dilemma).
Declares the minimum events any runtime SHOULD emit per skill
invocation. The protocol declares the **event surface**; runtimes pick
the transport (stdout, OpenTelemetry, custom dashboard, etc.).

Minimum event surface:

| Event | Fields | When |
|---|---|---|
| `skill.invoked` | skill_id, invoker, inputs (redacted), timestamp | At start |
| `skill.completed` | skill_id, outputs (paths), duration_ms, status | On finish |
| `skill.error` | skill_id, error_class, message, timestamp | On failure |
| `memory.write` | path, skill_id, bytes, timestamp | Per write |
| `integration.called` | integration_id, operation, skill_id, duration_ms | Per call |
| `verifier.run` | verifier_skill_id, gated_skill_id, verdict, timestamp | When applicable |
| `cache.hit` / `cache.miss` | prompt_prefix_hash, skill_id, savings_pct | Per LLM call |

Runtimes emit these to whatever observability backend they choose.
Dashboards (the Claude Platform one or any equivalent) consume the same
event surface.

### 3.6 — `cache_affinity` declaration

Declares whether content is stable (cache aggressively) or dynamic
(expect frequent invalidation). Applies to context bundles, port
manifests, and HarnessBundle entries.

```yaml
bundles:
  - id: soul
    cache_affinity: stable
    # ...
  - id: morning-feed
    cache_affinity: dynamic
    # ...
```

Default: `stable`. Runtimes implementing prompt caching use this to
inform cache eviction strategy.

### 3.7 — `load_mode` on Integration

Declares whether the runtime should eagerly load the integration's
schema/handles or pull them on demand. Maps to the Claude Platform "tool
search" optimization.

```yaml
- id: salesforce
  load_mode: on_search
  # ...
- id: linear
  load_mode: eager
  # ...
```

Default: `eager`. Setting `on_search` declares the integration's schema
is large enough that lazy loading is worthwhile.

### 3.8 — `intermediate` on Skill outputs

Marks outputs that exist purely for downstream tool consumption — they
don't need to land in the operator's context window. Maps to
"programmatic tool calling" optimization.

```yaml
outputs:
  - name: query_results_raw
    type: object
    intermediate: true
  - name: summary_report
    type: file_path
    write_to: "outputs/report.md"
```

Default: `false`. Setting `true` declares the output is internal scaffolding.

### 3.9 — `compactable` on MemoryMap folders

Declares whether a folder's contents can be summarized/dropped from
in-session context after writing. Content persists on disk; just doesn't
need to stay live in the agent's context window.

```yaml
structure:
  - path: "raw/"
    compactable: true
  - path: "wiki/clients/"
    compactable: false
```

Default: `false`. Setting `true` declares the folder is a buffer/staging
area whose in-session presence after write is unnecessary.

### 3.10 — `model_tier` + `advisor_escalation` on Skill

Declares which skills warrant the more expensive model (Karpathy + Claude
Platform advisor strategy).

```yaml
- id: prospect-qualification
  model_tier: standard
- id: gamut-audit
  model_tier: advanced
- id: engagement-delivery
  model_tier: standard
  advisor_escalation: gamut-audit
```

`model_tier: standard` — defaults to the runtime's standard model.
`model_tier: advanced` — runtime escalates to its premium tier
(currently Opus).

`advisor_escalation` — optional skill_id that the skill can call out to
when it needs a one-shot advisor judgment. The runtime invokes the
advisor skill (which itself runs on the advanced model) and returns the
verdict into the calling skill's context.

## 4) Manifest schema extension

Bump `protocol_version` pattern stays `^1\.\d+\.\d+$` (already accepts
1.4.0). Add the new optional fields to existing `$defs`:

- `$defs.skill` — add `sponsors`, `verifier_skill_id`, `multiplayer`,
  `model_tier`, `advisor_escalation`, `cache_affinity`
- `$defs.skillOutput` — add `intermediate`
- `$defs.integration` — add `load_mode`, allow `scope` items to be
  string OR object with `operation` + `direction`
- `$defs.memoryFolder` — add `compactable`
- `$defs.contextBundle` — add `cache_affinity`
- `$defs.runtimePorts` — add `cache_affinity` to port refs

All additions are optional. Existing v1.0.0–v1.3.0 manifests parse
cleanly with no changes.

## 5) Conformance — fully optional

No upgrade pressure. The 10 additions are **declarative hints**, not
requirements. Runtimes that don't implement caching/compaction/advisor
strategies ignore them; runtimes that do benefit from the declared
properties.

Two exceptions where conformance is SHOULD rather than MAY:

- **Audit events (§3.5)** — SHOULD be emitted by any conforming runtime.
  Different transport, same surface. Required for any tenant who wants
  reproducible operational visibility.
- **`sponsors[]` (§3.1)** — SHOULD be populated on Validated-evidence
  projects (cross-references the existing evidence-level conformance
  pattern).

## 7) Open dilemmas — RESOLVED 2026-05-12

All 8 locked at recommended defaults:

| # | Resolution |
|---|---|
| 7.1 | `cache_affinity` defaults to **stable** |
| 7.2 | `model_tier` is enum **`standard \| advanced`** |
| 7.3 | `advisor_escalation` is a **skill_id** reference |
| 7.4 | Audit events are **SHOULD** for v1.4.0 conformers |
| 7.5 | Audit-event spec lives in **new PROTOCOL.md §12** |
| 7.6 | `multiplayer` defaults to **false** (opt-in) |
| 7.7 | `intermediate` defaults to **false** (visible) |
| 7.8 | **One PR** with all 10 additions stacked |

Detailed rationale for each below.

### 7.1 — `cache_affinity` default: stable or dynamic?

**Options:** (a) default `stable` (cache by default), (b) default
`dynamic` (cautious), (c) require explicit declaration.

**Recommendation:** (a) default `stable`. Most protocol artifacts (soul,
identity, purpose, taste) are stable by construction — that's why they're
governance artifacts. Tenants opt into `dynamic` for live feeds. Lowest
friction, highest cache-hit-rate-by-default.

### 7.2 — `model_tier` values: enum or model name?

**Options:** (a) two-tier enum `standard | advanced`, (b) model names
(`sonnet`, `opus`), (c) free-form string.

**Recommendation:** (a) enum. Tier abstracts away the specific model
name, which evolves faster than the protocol. Runtimes map
`advanced` → whatever their premium tier is at the time. Asana's
Managed Agents work this way.

### 7.3 — `advisor_escalation`: skill_id (composable) or model name (direct)?

**Options:** (a) skill_id — escalation goes through a registered Skill
(composable, auditable), (b) model name — direct escalation to a
different model.

**Recommendation:** (a) skill_id. The skill abstraction means escalation
gets its own audit trail, can have its own context bundle, and can be
verified like any other skill. Going to a raw model name is faster but
loses the protocol's existing accountability surface.

### 7.4 — Audit events: SHOULD or MAY?

**Options:** (a) SHOULD for any conforming v1.4.0 runtime, (b) MAY,
(c) hold off until v0.0.9 when there's more runtime experience.

**Recommendation:** (a) SHOULD. Observability is the missing layer
mentioned in every research source. Specifying the surface (without
specifying transport) is the right level of commitment. Runtimes that
can't emit events stay v1.3.0 conformant; v1.4.0 runtimes commit.

### 7.5 — Where does the audit-event spec live in PROTOCOL.md?

**Options:** (a) add to §9 (Runtime Ports — observability is a runtime
concern), (b) new §12 (Observability Interface), (c) inline in §9.3
(Skills).

**Recommendation:** (b) new §12. Audit events span all four ports
(invocations, memory writes, integration calls, verifier runs). A
dedicated section keeps the cross-cutting concern visible. ~40–60 lines.

### 7.6 — `multiplayer` default: true or false?

**Options:** (a) default `false` (single-user assumption is the safer
baseline), (b) default `true` (multi-user is the realistic case for any
operating tenant world).

**Recommendation:** (a) default `false`. Multi-user memory accrual has
real consequences — corrections from user A propagate to user B. That's
an explicit opt-in, not an implicit default. Tenants who want it
declare it.

### 7.7 — `intermediate` default: false (visible) or true (hidden)?

**Options:** (a) default `false` (visible in context — safer), (b)
default `true` (efficient by default).

**Recommendation:** (a) default `false`. The protocol's pattern is that
output is operator-visible unless explicitly marked otherwise. Tenants
opt into hiding intermediate outputs.

### 7.8 — Scope: one PR with everything, or split into Cluster A + Cluster B?

**Options:** (a) one PR — everything is small additive schema changes,
(b) two PRs — Cluster A (semantics) then Cluster B (efficiency).

**Recommendation:** (a) one PR. All 10 items are additive optional
fields touching the same schema. Split would create cross-references
between v0.0.8 and v0.0.8.1 that obscure the unified intent. Smaller
than v0.0.5/v0.0.6/v0.0.7 each were individually.

## 11) Risks

- **Hint creep.** Adding 10 optional fields obscures the spec for
  readers focused on the core mandatory surface. Mitigation: keep the
  prose tight; group fields by purpose in the spec; clearly mark
  defaults inline.
- **Runtime drift on hint interpretation.** Different runtimes will
  interpret `cache_affinity: stable` and `compactable: true`
  differently. Mitigation: explicit prose on intent; declare that
  interpretation is runtime-side; revisit in v0.1.0 if drift creates
  interoperability problems.
- **Premature optimization.** Some of these efficiency hints may be
  answers to questions the protocol shouldn't ask. Mitigation: all
  fields are MAY/SHOULD, never MUST. v0.1.0 compression review can
  remove unused fields.
- **Audit event scope creep.** §12 could easily grow into a full
  observability spec. Mitigation: keep §12 to the seven-event minimum
  surface; defer richer schemas to a separate v0.0.9+ doc if needed.
- **`scope` polymorphism (string OR object).** Allowing both forms in
  v1.4.0 is a one-time compatibility hack. Future v0.1.0 could collapse
  to object-only. Mitigation: clearly document the dual shape in §9.6
  with a deprecation note for v0.1.0.

## 12) What this unlocks

- **Runtime cost predictability.** Tenants can declare cache affinity
  and intermediate outputs; runtimes that respect these hints get the
  $30.57 → $5 efficiency the Claude Platform talk showed.
- **Verifier pattern generalized.** The design-eye-evaluator becomes a
  template. Any output-producing skill can declare a verifier. New
  verifier skills for non-design outputs (legal, accuracy, policy)
  become first-class.
- **Multiplayer tenant worlds.** Asana-style shared memory becomes a
  declarable property. Runtimes can choose which skills are single-user
  vs. multi-user without protocol changes per case.
- **Observability without a dashboard.** Facework specifies the event
  surface; any dashboard implements it. The Claude Platform dev
  dashboard (or Corey's runtime dashboard, or a tenant's custom UI)
  consume the same events.
- **Advisor escalation as a first-class skill pattern.** Calling out to
  a more expensive model for one-shot judgment becomes a declarable
  workflow, not an undocumented runtime trick.
