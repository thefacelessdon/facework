### Stability

**What didn't:** Playbooks were written without validating against real workflows ("it varies"). They're prescriptive, not descriptive.

## Protocol-Level Issues (Must Fix)

These issues are not HUE-specific. They will break every future protocol run.

### Issue 1: No Phase 0 Intake Skill

**Problem:** The protocol has no structured intake process. There's no `/fw-intake` or Phase 0 skill that:
- Confirms which project/directory the protocol targets
- Scans for existing artifacts (brand docs, specs, previous work)
- Understands the team (who, roles, capacity)
- Identifies what already exists vs what needs to be created
- Produces a project context summary before Phase 1 begins

**Impact:** Frequency re-asked questions the restructure notes already answered. The brand toolkit wasn't discovered until the user mentioned it. Working directory confusion caused false starts.

**Fix:** Create `/fw-intake` as Phase 0. It runs before everything else. It reads the repo, asks targeted questions, and produces a context document that all subsequent phases reference.

### Issue 2: Phase Sequence is Implicit

**Problem:** The CLAUDE.md lists phase names in order but doesn't number them. The skill descriptions overlap — both `fw-frequency` and `fw-semantics` say "before anything gets built." Nothing maps the sequence to numbered phases with entry/exit criteria.

**Impact:** The protocol runner defaulted to list order (frequency first) when the actual sequence starts with semantics. This skipped Phase 1 entirely.

**Fix:**
- Add explicit phase numbers to every skill description: "Phase 0: Intake", "Phase 1: Semantics", etc.
- Add entry criteria: "Requires: Phase 0 context document"
- Add exit criteria: "Complete when: [specific deliverables] exist"
- Update CLAUDE.md with the numbered sequence

### Issue 3: Skill Descriptions Overlap

**Problem:** `fw-frequency` says "before anything gets designed or built." `fw-semantics` says "before product framing." Both claim the same territory. A protocol runner can't tell which comes first from the descriptions alone.

**Impact:** Wrong phase was run first. Semantics was skipped.

**Fix:** Each skill description should state its position clearly:
- "This is Phase N of the Facework protocol. It runs after Phase N-1 and before Phase N+1."
- Remove overlapping "do this first" language. Only Phase 0 (Intake) should say "before everything."

### Issue 4: No Existing Artifact Ingestion

**Problem:** Frequency's discovery step asks the user questions as if starting from zero. It doesn't read existing docs first. If a project already has a business model draft, restructure notes, brand toolkit, or strategic docs — Frequency should ingest them before asking.

**Impact:** Wasted user time re-answering questions that were already documented. Made the process feel redundant rather than building on existing work.

**Fix:** Add to Frequency (and every phase): "Step 0: Read existing artifacts. Scan the project directory for docs, specs, designs, and prior work. Ingest before asking." Each phase should build on what exists, not start from scratch.

### Issue 5: Build Sequence vs Prototype Tension

**Problem:** Current produced Decision 006 ("encode first, then build site"). Resonance then built the site. This was reconciled in-session but the tension exists in the methodology — Resonance says "build a working prototype" which can conflict with a "don't build yet" decision.

**Impact:** Confusion about whether the prototype violates the decision. Required in-session justification.

**Fix:** Resonance skill should explicitly address this: "If a previous decision says 'don't build X yet,' the prototype demonstrates the architecture and data model without building the final X. The prototype is scaffolding, not the product."

## Coherence Assessment

**Structural coherence:** The HUE system holds its shape. Specs are internally consistent. Decisions don't contradict each other. Ownership model is enforced at the schema level. A new builder can start from the handoff package. **Score: Strong.**

**Cultural coherence:** The system names its energy sources, checks for extraction, and writes governance for participants — not just the platform. But it hasn't been validated by the community (talent, clients, cultural communities). The taste encoding hasn't run. The site has no real imagery. **Score: Designed for, not yet proven.**

**Process coherence:** The Facework protocol produced good output despite its own structural gaps. But it wasted time, confused sequence, and missed existing context. The protocol needs the same treatment it gives to projects — extraction check on its own friction. **Score: Needs work.**
